5 Essential Secure Coding Snippets Every Web Developer Should Know

Protect your applications from common vulnerabilities like XSS, SQL Injection, and more with these practical, easy-to-implement code examples.

Published on October 27, 2025 • By the iCraftNow Team • 8 min read

As web developers, we love to build features. But in the race to ship, security can sometimes become an afterthought. This is a dangerous mindset. A single vulnerability can lead to data breaches, loss of user trust, and significant financial damage. The good news is that preventing the most common attacks often comes down to a few key coding practices.

In this post, we'll share five essential code snippets that address critical web security vulnerabilities. Think of these as your first line of defense—easy to understand, simple to implement, and powerful in their protection.

1. Sanitize Inputs to Prevent Cross-Site Scripting (XSS)

The Threat: XSS attacks occur when a malicious script is injected into a trusted website. If your site displays user-provided content (like comments or profile names) without sanitizing it, an attacker could inject a <script> tag that steals other users' session cookies or performs actions on their behalf.

The Solution: Always escape HTML content before rendering it on a page. This turns special characters like < and > into their safe entity equivalents (e.g., &lt; and &gt;), so the browser treats them as text, not code.


function escapeHTML(str) {
    const p = document.createElement('p');
    p.appendChild(document.createTextNode(str));
    return p.innerHTML;
}

// Usage:
const userInput = '';
const sanitizedInput = escapeHTML(userInput);
// document.body.innerHTML = sanitizedInput; // This is now safe

Note: While this simple function works for client-side examples, most modern frontend frameworks (like React and Vue) automatically sanitize content. On the backend, use a trusted library like DOMPurify for robust sanitization.

2. Use Parameterized Queries to Stop SQL Injection

The Threat: SQL Injection happens when an attacker manipulates a SQL query by inserting their own malicious SQL. This can be used to bypass logins, steal an entire database, or even delete data. It's one of the oldest and most devastating web vulnerabilities.

The Solution: Never, ever build SQL queries by concatenating strings with user input. Instead, use parameterized queries (also known as prepared statements). This separates the SQL command from the data, making it impossible for user input to be executed as code.


// Using the 'mysql2' package in Node.js
const mysql = require('mysql2/promise');

async function getUser(email) {
    const connection = await mysql.createConnection(/* config */);

    // INSECURE: Vulnerable to SQL Injection
    // const [rows] = await connection.execute(`SELECT * FROM users WHERE email = '${email}'`);

    // SECURE: Using a parameterized query
    const sql = 'SELECT * FROM users WHERE email = ?';
    const [rows] = await connection.execute(sql, [email]);
    
    return rows[0];
}

3. Hash and Salt Passwords Correctly

The Threat: If your database is ever compromised, storing passwords in plain text is a catastrophic failure. Even simple hashing like MD5 or SHA1 is no longer secure due to rainbow tables and brute-force attacks.

The Solution: Use a strong, slow, and salted hashing algorithm like Bcrypt or Argon2. These algorithms are computationally expensive, which makes them resistant to brute-force attacks. "Salting" adds a unique, random string to each password before hashing, ensuring that two identical passwords result in different hashes.


// Using the 'bcrypt' package in Node.js
const bcrypt = require('bcrypt');
const saltRounds = 10; // The higher the number, the more secure but slower

async function hashPassword(plainTextPassword) {
    const hash = await bcrypt.hash(plainTextPassword, saltRounds);
    return hash;
}

async function checkPassword(plainTextPassword, hash) {
    const isMatch = await bcrypt.compare(plainTextPassword, hash);
    return isMatch;
}

4. Set Secure HTTP Headers with Middleware

The Threat: By default, web servers don't send HTTP headers that can protect your users from common attacks like clickjacking, cross-site scripting, and protocol downgrade attacks.

The Solution: Use a middleware library to set several important security headers on every response. These headers instruct the browser to enforce stricter security policies.


// Using the 'helmet' package in Node.js with Express
const express = require('express');
const helmet = require('helmet');

const app = express();

// Use Helmet to set various security headers by default
app.use(helmet());

// Example headers set by Helmet:
// - Strict-Transport-Security: Enforces secure (HTTP over SSL/TLS) connections.
// - X-Content-Type-Options: Prevents MIME-sniffing.
// - X-Frame-Options: Provides clickjacking protection.
// - X-XSS-Protection: Enables the XSS filter built into most modern browsers.

5. Protect Against Cross-Site Request Forgery (CSRF)

The Threat: A CSRF attack tricks a logged-in user into submitting a malicious request to your web application. For example, a user might click a link on a malicious site that, in the background, makes a request to `yourbank.com/transfer?to=attacker&amount=1000`, using the user's active login cookie.

The Solution: The most common defense is the Synchronizer Token Pattern. The server generates a unique, random token for each user session and embeds it in a hidden form field. When the user submits the form, the server checks that the submitted token matches the one stored in the session. An attacker cannot guess this token, so their forged request will fail.


// Using the 'csurf' package in Node.js with Express and sessions
const express = require('express');
const session = require('express-session');
const csrf = require('csurf');

const app = express();

// 1. Set up sessions
app.use(session({ secret: 'your-secret-key', /* ... */ }));

// 2. Set up CSRF protection
const csrfProtection = csrf({ cookie: true });
app.use(csrfProtection);

// 3. In your route, pass the token to the view
app.get('/profile', (req, res) => {
  res.render('profile', { csrfToken: req.csrfToken() });
});

// 4. In your form (e.g., in an EJS/Pug template):
// <form action="/profile/update" method="POST">
//   <input type="hidden" name="_csrf" value="<%= csrfToken %>">
//   <!-- ... other form fields ... -->
// </form>

Conclusion: Security is a Process

Writing secure code isn't about memorizing a thousand tricks. It's about understanding the common threats and consistently applying fundamental principles. By integrating these five practices into your development workflow, you can build a strong foundation of security for any web application. Start with these, stay curious, and never stop learning.
